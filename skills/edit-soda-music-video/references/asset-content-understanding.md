# 执行模型的素材理解与语义匹配

## 原则

素材内容理解和素材匹配不通过脚本调用另一个大模型，也不配置 API、模型端点或向量库。执行这个 Skill 的模型本身就是理解和匹配模型，直接使用可用的 Read/图像查看工具检查素材，再编辑工作区 Manifest 和时间轴。

`sync-assets` 只负责确定性的文件扫描、类型判断和 FFprobe 元数据。它不会理解图片，也不会根据文件名替素材做决定。

生成视频前必须完成全量视觉素材理解门禁：Manifest 中所有 `kind=image` 或 `kind=video` 的记录都必须有非空 `description` 和源像素坐标的 `effective_region`，Manifest 的 `asset_root` 必须与当前任务一致，时间轴引用的 logo、尾帧和普通素材都必须出现在 Manifest 中。`preflight`/`render` 会检查这些条件；任一条件不满足时先停止渲染，完成下面的 Read 流程后再继续。

## 入库时的操作顺序

1. 运行 `sync-assets`，读取工作区 Manifest；
2. 查看 Manifest 的 `changes.added`、`changes.modified`，以及缺少 `description` 或 `effective_region` 的素材；若要进入正式生成，不能只理解变化项，必须把所有图片/视频都核对为已有准确描述和有效区域；
3. 对每张待理解的图片使用 Read 工具打开原图，逐张确认真实画面；
4. 对每段待理解的视频先用 FFprobe 获取时长，再用 FFmpeg 导出首帧、四分之一、中点、四分之三和尾帧等缩略图；使用 Read 工具逐张查看这些缩略帧，不能只看文件名或第一帧；
5. 对透明 PNG 必要时在浅色和深色背景上各查看一次，区分透明留白与实际内容；
6. 为每个素材按下面的强制字段覆盖规则写一段准确、具体、可检索的中文 `description`；不能只概括素材主题。同时识别真正承载画面信息的最小矩形联合区域，以源文件像素坐标写入 `effective_region`；
7. 透明留白、纯色空白边距和无内容画布不进入 effective_region；人物、图标、文字、UI、手机框、按钮及有表达作用的背景进入 effective_region。视频使用代表帧中有效内容的联合边界；
8. 只把 `description` 和 `effective_region` 写入对应 Manifest 素材记录，不增加 `keywords`、`recommended_usage`、向量或模型调用字段；
9. 保存 Manifest 前逐项执行 description 质量自检，确认页面类型、结构数量、关键文字/数字、状态和可表达语义都已写清，且全部来自实际画面；不能把文件名、目录名或粗分类当成画面事实；
10. 再运行 `preflight --asset-manifest ...`，确认门禁报告 `asset_understanding.ok=true` 后才允许生成视频。

## description 写法

`description` 必须基于 Read 实际看到的画面，通常用 `2–4` 句具体中文覆盖以下适用信息，不能只写一个宽泛主题。只有内容单一的图标、logo 或纯装饰素材可以用一句具体描述；手机 UI、金额、权益和操作录屏必须使用 `2–4` 句：

1. **素材和页面类型**：说明是完整页面、弹窗、结果页、卡片、图标、照片还是录屏，以及具体页面用途；
2. **主体与布局结构**：说明画面主体、区域排列，以及账户、分组、卡片、按钮、档位或人物数量；
3. **关键可见文字和数字**：逐项写出与检索、功能或利益点有关的清晰文字、金额、时长、次数、比例和状态值；
4. **动作或当前状态**：说明正在操作什么、当前选中/未选中/成功/失败/可用/禁用状态，静态页面则说明当前展示阶段；
5. **可直接表达的口播语义**：只写画面本身能够支持的功能或利益点，不把推测当成事实；
6. **必要的排除说明**：容易与相似素材混淆时，说明它不是哪类页面，例如“不是提现成功结果页”“不是单纯余额卡片”。

对于手机 UI、提现、金额、会员、金币、奖励和权益类素材，上述第 1–5 项为强制项。相同按钮或档位重复出现在多个账户/分组时，必须写明分组数量，并完整列出每组清晰可见的关键选项；不能用“小额金额”“多个档位”“展示功能”等概括词代替实际数字和结构。

数字和文字必须按画面逐字记录。局部文字模糊但不影响素材身份时，明确写“部分文字不可辨认”；如果连页面类型、关键金额或主要状态都无法确认，则保留空 `description` 并标记待人工确认，不得根据文件名补全。

示例：

```json
{
  "relative_path": "正文端内物料/通用/模式1.mp4",
  "kind": "video",
  "category": "ui_operation",
  "description": "这是一段手机录屏，展示用户进入汽水音乐播放设置页面并切换播放模式。画面中能看到循环播放和随机播放等选项，当前处于播放模式选择阶段，适合表达播放模式或播放设置功能。",
  "effective_region": {
    "x": 0,
    "y": 0,
    "width": 1080,
    "height": 1920,
    "coordinate_space": "source_pixels"
  }
}
```

提现档位页面示例：

```json
{
  "description": "完整的提现页面截图，顶部显示当前余额 8.72 元。页面下方有两组提现账户，每组都包含 0.3 元、15 元和 30 元三个提现档位，这是提现档位选择页而不是提现成功结果页。该画面能够直接表达满 0.3 元即可提现的利益点。"
}
```

不要写成“一个 App 页面”“一个功能截图”“展示提现功能界面和小额金额显示”等空泛描述。不要凭文件名臆测，也不要把透明区域描述成黑色背景。

## description 写回前自检

每条视觉素材保存到 Manifest 前逐项确认：

- 是否写明了素材或页面的具体类型；
- 是否写明了主体、布局、分组和数量；
- 是否逐项记录了与功能或利益点有关的关键文字和数字；
- 是否说明当前动作、阶段、选中状态或结果状态；
- 是否写明画面能够直接支持的口播语义；
- 是否需要加入“不是结果页/不是余额卡片”等排除说明；
- 是否存在只凭文件名、目录名或类别推断的内容；
- 是否仍含“小额金额”“多个按钮”“展示功能”等可被真实画面信息替换的泛化表达。

只要某项适用于当前素材但没有写清，就先重新 Read 原图或代表帧并重写 description，不能直接进入素材匹配。视频还必须核对多张代表帧，把过程中出现的页面变化、操作步骤和最终状态写入 description，不能只描述第一帧。

例如一个 1080×1920 的驾车图标文件，只有中心 `x=440,y=860,width=200,height=200` 的白色图标有内容，其余区域为空白或透明，则写：

```json
"effective_region": {
  "x": 440,
  "y": 860,
  "width": 200,
  "height": 200,
  "coordinate_space": "source_pixels"
}
```

渲染器只用该 200×200 区域判断是否遮挡品牌区域，不能因为 1080×1920 源画布越界而移动或缩小素材。

## 什么时候重新查看

- 新增素材：必须查看并写 description 与 effective_region；
- Manifest 标记为 modified：源文件可能被替换，必须重新查看；
- 已有 description/effective_region 且素材未变化：复用，不重复查看；
- 只有文件名或目录名变化、画面内容未变：可以保留原理解结果，但要检查路径是否仍对应同一素材；
- 不能确认素材内容或有效区域：对应字段留空并在报告中标记待人工确认，不得编造。

## 根据口播匹配素材

执行模型先把当前口播逐句分为“利益点”和“非利益点”。只有明确表达产品功能带来的用户价值、合规权益、已授权优惠或具体使用收益的句子才属于利益点。寒暄、开场过渡、情绪铺垫、泛泛描述、单独 CTA 和免责声明不触发素材匹配。

只对利益点选材，并先检查 [special-material-matches.md](special-material-matches.md)。特殊规则的判断由执行模型完成，不通过脚本理解台词；一旦命中，跳过普通候选排序，优先使用参考素材或由 Read 确认核心画面完整的轻度变体。未命中特殊规则时，才读取 Manifest 的 `description` 做普通语义匹配：

1. 非利益点直接保持数字人主画面，不查询、不选择、不插入普通素材；logo、警示语和官方尾帧不属于这里的普通素材；
2. 对利益点先判断是否命中特殊规则；命中时写入参考素材或合格变体的真实 `path` 和 `special_match_rule`，不得改选语义不同的余额卡片、结果页或局部截图；
3. 未命中特殊规则时，按 `kind`、`category`、布局、时长、合规状态和黑边检查做硬过滤；
4. 对剩余素材逐条阅读 description，并与利益点句子的主体、动作、功能、收益和场景进行语义比较；
5. 选择真正能解释这条利益点的素材，不因为关键词相似就选无关画面；
6. 将选中的真实 `path` 写入时间轴，同时写入 `semantic_role: benefit_point`、`matched_benefit_text`、素材的 `name`、`category`、`kind`、`layout` 和时间范围；
7. 没有准确匹配时明确报告“没有合适素材”，不要强行填充；
8. 如有两个以上普通候选，记录选择理由和未选原因到任务报告，方便人工复核。

模型不能创造 Manifest 中不存在的路径，也不能绕过现有的 logo/警示语保护区、固定黑边、渠道合规和图层顺序规则。
